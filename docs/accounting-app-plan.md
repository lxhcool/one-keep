# OneKeep 记账 App 规划（v0.1）

## 1. 目标与边界

- 目标：先做可用的个人记账 MVP，确保录入快、统计准、数据不丢。
- MVP 不做：复杂预算体系、多人协作、企业级审批流。
- 设计原则：先本地可用，再云端同步；先核心流程，再管理后台。

## 2. 是否需要后台管理端

结论：`MVP 阶段不需要后台管理端`，所有管理能力放在 App 内即可。

建议加后台的触发条件：

- 需要运营人员在网页端批量管理分类/模板/公告。
- 需要审计日志、角色权限、客服代操作。
- 需要面向多租户或 B 端客户的运营能力。

在个人记账或小规模 C 端早期，这些通常不是刚需。

## 3. 服务端是否必需

### 方案 A：无服务端（推荐起步）

- 架构：Flutter + 本地数据库（SQLite/Drift）。
- 优点：开发快、成本低、离线可用、复杂度最低。
- 风险：换机迁移和多端同步弱，需要手动导出导入。

### 方案 B：轻量服务端（推荐第二阶段）

- 架构：Flutter + API 服务 + PostgreSQL。
- 用途：账号体系、云备份、多设备同步、数据恢复。
- 优点：体验完整，可持续演进。
- 代价：增加部署、监控、数据一致性复杂度。

## 4. 服务端技术建议（不局限 NestJS）

优先级建议：

1. `Fastify + Prisma + PostgreSQL`
2. `NestJS + Prisma + PostgreSQL`
3. `Supabase（BaaS）`

选择建议：

- 你想要“轻量、快交付”：选 Fastify。
- 你想要“强约束、模块化、团队协作”：选 NestJS。
- 你想要“最少后端代码”：选 Supabase。

## 5. 客户端技术栈（Flutter）

- 状态管理：`Riverpod`
- 本地存储：`Drift(SQLite)`
- 路由：`go_router`
- 网络层（有服务端时）：`dio`
- 数据模型：`freezed + json_serializable`
- 图表：`fl_chart`

## 6. 分阶段路线图

### Phase 1（MVP，2-4 周）

- 账单 CRUD（收入/支出/转账）
- 分类与账户管理
- 月/周统计图
- 本地备份导出（JSON）

### Phase 2（增强，2-4 周）

- 登录注册
- 云端备份与恢复
- 多设备同步（增量同步）

### Phase 3（可选，2-3 周）

- Web 后台（Vue3 或 React 二选一）
- 运营配置和审计日志

## 7. 后台前端选型（若进入 Phase 3）

- 偏管理系统和工程规范：`React + Ant Design`
- 偏上手速度与模板生态：`Vue3 + Element Plus`

若你个人更熟悉其一，优先“熟悉度”而不是框架优劣。

## 8. 依赖策略

- 当前仓库保持最小依赖（根目录不引入业务依赖）。
- 仅在创建 `client/server/admin` 子项目时再分别安装依赖。
- 原则：每个子项目独立依赖，避免把业务包装到根目录。
