# 服务端实现

## 本轮目标

- 基于 FR-001 ~ FR-007 形成登录、注册、微信登录服务端实施方案。
- 明确账号识别、唯一性校验、错误码、登录态签发和实现顺序。
- 输出后续开发所需的接口落地与测试清单。

## 接口与数据改动

| 需求编号 | 接口/模块 | 改动内容 | 当前状态 | 备注 |
| --- | --- | --- | --- | --- |
| FR-001, FR-002, FR-006 | /api/auth/login | 新增账号密码登录接口，支持用户名或手机号 account | 方案已定义 | 登录成功返回 user + accessToken |
| FR-003, FR-004, FR-005, FR-006 | /api/auth/register | 新增注册接口，创建用户并返回注册结果 | 方案已定义 | 是否直接签发登录态待确认 |
| FR-007 | /api/auth/wechat | 预留微信登录接口 | 方案已定义 | 可按开关能力后置实现 |
| FR-004, FR-005 | /api/auth/check-availability | 新增用户名/手机号可用性校验接口 | 方案已定义 | 可选接口，提交校验仍为准 |

## 推荐模块拆分

| 模块 | 职责 | 关联需求 | 备注 |
| --- | --- | --- | --- |
| auth-service | 登录、注册、登录态签发 | FR-001, FR-002, FR-003 | 核心认证服务 |
| account-validator | 手机号、用户名、昵称、密码规则校验 | FR-004, FR-005 | 保证前后端规则统一 |
| user-repository | 查询/创建用户与唯一性检查 | FR-003, FR-004 | 需支持 username 与 phone 双索引 |
| token-service | accessToken / refreshToken 签发与校验 | FR-002 | refreshToken 是否首发启用待确认 |
| wechat-auth-adapter | 微信授权码换取用户身份 | FR-007 | P1，可延后实现 |

## 实现步骤建议

| 步骤 | 动作 | 产出 | 风险控制 |
| --- | --- | --- | --- |
| 1 | 冻结字段规则和错误码 | 注册/登录字段规范与错误枚举 | 避免客户端提示反复调整 |
| 2 | 完成 /api/auth/register | 新用户创建闭环 | 优先明确唯一性冲突返回 |
| 3 | 完成 /api/auth/login | 老用户恢复登录闭环 | account 识别规则必须稳定 |
| 4 | 决定是否实现 /api/auth/check-availability | 更流畅的输入期校验 | 不实现也不能影响注册主流程 |
| 5 | 预留 /api/auth/wechat | 第三方接入边界清晰 | 避免首发被第三方能力阻塞 |

## 推荐数据模型

| 对象 | 关键字段 | 说明 |
| --- | --- | --- |
| UserAccount | id, phone, username, nickname, passwordHash, createdAt | 认证核心用户实体 |
| LoginSession | userId, accessToken, refreshToken?, expiredAt | 登录成功后的会话结构 |
| AvailabilityResult | phoneAvailable, usernameAvailable | 可用性校验结果 |
| WechatAuthResult | openId/unionId, isNewUser, linkedUserId? | 微信登录适配层结果 |
