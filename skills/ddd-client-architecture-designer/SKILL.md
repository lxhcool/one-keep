# 客户端 DDD 架构设计

聚焦客户端实现边界，将需求与领域规则落到可实现的前端架构。

## 工作流

### 1. 收集并约束输入

- 输入优先读取：`01-layout/*.md`、`02-requirements/*.md`、`03-thinking/*.md`。
- 识别客户端必须承载的业务规则与交互状态。
- 缺失信息登记为待确认项，不擅自扩展范围。

### 2. 设计客户端分层

- 定义展示层、应用编排层、领域表示层、基础设施层职责。
- 明确页面/组件/状态容器之间的依赖方向。
- 约束 UI 组件只承载展示与交互，不内聚业务规则。

### 3. 设计状态与数据流

- 定义页面级与跨页面状态模型。
- 说明加载、空态、错误、重试、回滚等状态转换。
- 设计本地缓存、离线回放、同步冲突处理策略。

### 4. 定义客户端契约适配

- 定义 DTO 到 ViewModel 的映射边界。
- 定义 API 异常到用户可见反馈的转换规则。
- 标注版本兼容策略与灰度字段兜底策略。

### 5. 输出架构文档

- 使用 `references/client-architecture-template.md`。
- 输出需可直接指导客户端实现与代码结构落地。

## 质量标准

- 客户端职责清晰，不侵入服务端业务不变式。
- 状态流闭环，异常路径可验证。
- 文档可追踪到需求编号与页面场景。

## 工作流输出路径

- 提供工作流目录时，输出写入：
- `docs/workflows/<workflow-id>/04-architecture/client-architecture.md`
