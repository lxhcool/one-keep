# 服务端优先的实现与联调

基于需求文档完成服务端接口实现、客户端页面绘制与接口对接，并对结果做最小自检。

## 适用场景

- 需求文档已经确认，可以开始开发。
- 需要先完成服务端能力，再进行客户端绘制和联调。
- 需要在开发同时沉淀变更记录、测试结果和剩余风险。

## 工作流

### 1. 建立追踪基线

- 将功能范围映射到需求编号与页面分析结果。
- 确认本轮改动涉及的服务端模块、客户端模块和外部依赖。
- 开发前冻结非目标范围，避免需求蔓延。

### 2. 先实现服务端

- 优先完成服务端业务规则、数据模型、接口与校验逻辑。
- 明确接口输入输出、错误码、权限要求和兼容性。
- 若存在迁移、初始化数据或异步任务，一并记录执行要求。

### 3. 再实现客户端

- 根据已确认的服务端契约完成页面绘制、状态处理和接口对接。
- 对齐加载、空态、错误态、成功态与重试逻辑。
- 若 UI 稿与需求有冲突，先记录并回写到文档，不擅自扩展。

### 4. 联调与自检

- 验证关键主流程和至少一条异常流程。
- 检查服务端返回与客户端展示是否一致。
- 补充最小必要的单元、集成或手动验证记录。

### 5. 输出实现文档

- 使用 `references/implementation-checklist.md` 作为默认模板。
- 输出至少包含：
- 实现计划
- 服务端改动摘要
- 客户端改动摘要
- 联调结果
- 最小测试清单
- 风险与后续事项

## 质量标准

- 顺序固定为先服务端、再客户端。
- 每项实现都能映射到需求编号。
- 文档需要说明已完成、未完成和待确认项。
- 测试可以轻量，但不能缺失。

## 输出路径

- 单独执行时，默认写入：
- `docs/<task-id>/03-development/implementation-plan.md`
- `docs/<task-id>/03-development/change-summary.md`
- `docs/<task-id>/03-development/test-notes.md`
- 串联执行时，沿用同一 `docs/<task-id>/` 目录。
