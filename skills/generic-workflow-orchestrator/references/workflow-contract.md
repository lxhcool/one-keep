# 通用工作流契约

适用于任意业务域和技术栈的三阶段交付链路。

## 1. 适用范围

- 面向从设计输入到需求文档，再到开发联调的通用流程。
- 不绑定具体行业词汇，业务名词以当前项目术语为准。
- 每个阶段可以单独执行，也可以被完整工作流串联。

## 2. 统一输入契约

- 必填输入：
  - 本轮目标
  - 目标页面或模块清单
- 可选输入：
  - 设计稿、原型、现有代码、历史文档、技术约束
- 缺失输入处理：
  - 允许带假设推进，但必须落盘到 `99-decisions/open-questions.md`

## 3. 统一输出契约

- 所有阶段产物写入：`docs/<task-id>/`
- 每阶段至少产出 1 份可被下阶段直接消费的文档
- 文档必须包含：输入来源、关键结论、待确认项、下一步动作

## 4. 统一追踪规则

- 需求编号：`FR-*` / `NFR-*` / `AC-*`
- 变更项编号：`IMP-*`
- 编号跨阶段保持稳定，禁止重编号
- 页面分析中的功能点需要能追踪到需求文档和开发记录

## 5. 阶段退出条件

- UI 分析阶段：页面结构、功能点、组件拆分、状态说明齐全
- 需求阶段：服务端需求先完成，客户端需求补全，验收标准可测试
- 开发阶段：服务端已实现，客户端已对接，最小测试和风险已记录

## 6. 决策与风险沉淀

- 关键决策追加到：`99-decisions/decision-log.md`
- 未决问题追加到：`99-decisions/open-questions.md`

## 7. 术语一致性

- 同一实体在全链路使用同一术语
- 不在技能中硬编码具体项目或行业名称
