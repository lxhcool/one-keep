# UI 布局分析

将视觉设计产物转换为开发可落地的结构化结果。

## 工作流

### 1. 收集输入

- 确认输入来源：截图、Figma 导出、设计说明、目标设备。
- 用户提供本地图片路径时，使用 `view_image` 查看。
- 对缺失信息先记录为假设，不阻塞分析。

### 2. 解析页面结构

- 识别页面区域：页头、导航、主内容、卡片区、弹窗、页脚。
- 标注主布局类型：单列、分栏、网格、瀑布流、叠层。
- 若可见响应式线索则直接标注，否则给出合理断点假设。

### 3. 划分组件边界

- 按“重复性、状态复杂度、可配置性”提取可复用单元。
- 区分页面容器与跨页面共享组件。
- 组件化判断规则：
- 同类结构复用 2 次以上，或行为/状态不简单时，提升为组件。
- 一次性且静态内容，保留为页面内结构。

### 4. 描述交互与状态

- 列出关键状态：空、加载、错误、成功、禁用、悬停、聚焦。
- 说明导航跳转、弹层开合、表单反馈等状态变化。
- 标注可访问性关键项：焦点顺序、键盘触达、对比度风险。

### 5. 输出结构化结果

- 使用 `references/output-template.md` 作为默认输出模板。
- 输出至少包含：
- 布局树
- 组件清单
- 交互/状态说明
- 待确认问题与假设

## 质量标准

- 术语与业务语言保持一致。
- 明确区分“观察结论”和“推断结论”。
- 组件命名可直接用于实现（例如 `ProductCard`、`FilterPanel`）。

## 工作流输出路径

- 提供工作流目录时，每个页面分析写入：
- `docs/workflows/<workflow-id>/01-layout/pages/<page-slug>.md`
- 跨页面组件汇总写入：
- `docs/workflows/<workflow-id>/01-layout/component-inventory.md`
