# UI 分析与组件拆分

将视觉输入转成后续需求与开发可直接消费的页面分析结果。

## 适用场景

- 用户提供截图、设计稿、原型、MasterGo 设计稿地址、Pencli 设计稿地址或页面链接，需要先看懂页面结构。
- 需要明确页面承载的功能点，而不仅仅是视觉布局。
- 需要判断哪些部分应拆成组件，以及哪些组件可以复用。
- 需要将分析结果落到 `docs` 目录，供后续需求与开发阶段使用。

## 工作流

### 1. 收集输入

- 确认输入来源：本地截图、Figma 导出、MasterGo 设计稿地址、Pencli 设计稿地址、设计说明、目标设备、已有页面链接。
- 用户提供本地图片路径时，使用 `view_image` 查看。
- 用户提供可访问的在线设计稿地址时，优先读取页面可见信息；若在线内容无法直接获取完整视觉信息，则要求补充截图或关键页面导出图。
- 缺失信息不阻塞分析，但必须记录为假设或待确认问题。

### 2. 解析页面与功能

- 识别页面区域：页头、导航、主内容、卡片区、弹窗、页脚。
- 标注主布局类型：单列、分栏、网格、瀑布流、叠层。
- 从页面元素反推用户可执行的功能点、主任务和关键路径。
- 将“观察事实”和“合理推断”分开记录，避免把猜测写成结论。

### 3. 划分组件边界

- 按重复性、状态复杂度、可配置性提取可复用单元。
- 区分页面容器组件与跨页面共享组件。
- 对每个候选组件执行项目内复用检查。
- 若找到可复用项，标记为“可匹配”，并记录组件路径与复用方式。
- 若未找到可复用项，标记为“需开发”，并写明不能复用原因。

### 4. 描述交互与状态

- 列出关键状态：默认、加载、空、错误、成功、禁用、选中。
- 描述导航跳转、弹层开合、表单反馈、分页筛选、危险操作确认。
- 若可见响应式线索则直接标注，否则给出断点假设。
- 补充可访问性关注项：焦点顺序、键盘触达、文案可读性、对比度风险。

### 5. 输出结构化结果

- 使用 `references/output-template.md` 作为默认模板。
- 文档默认优先使用表格表达，尤其是页面结构、功能点、组件分析、交互状态、待确认问题。
- 输出至少包含：
- 页面结构说明
- 功能点清单
- 布局树
- 组件清单与匹配检查
- 交互/状态表
- 待确认问题与假设表

## 质量标准

- 组件命名和功能命名可以直接被需求文档复用。
- 每个“需开发”组件都必须附带不能复用原因。
- 每个高风险交互都应写出异常流或未决点。
- 组件分析结果必须明确区分“可匹配”和“需开发”。
- 文档结果应让需求阶段无需重新做页面拆解。

## 输出路径

- 单独执行时，默认写入：
- `docs/<task-id>/01-ui-analysis/pages/<page-slug>.md`
- `docs/<task-id>/01-ui-analysis/component-inventory.md`
- 串联执行时，沿用同一 `docs/<task-id>/` 目录。
