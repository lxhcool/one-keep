# 产品需求编写

将页面分析结果与业务信息整理为 PM 可执行需求文档。

## 工作流

### 1. 统一输入

- 收集布局分析、业务目标、目标用户与约束条件。
- 将模糊目标改写为可衡量目标。
- 将未知项显式登记为假设。

### 2. 定义范围与用户价值

- 写清背景、用户问题、预期价值。
- 明确 in-scope 与 out-of-scope。
- 输出具备“角色-行为-价值”的用户故事。

### 3. 定义功能需求

- 按模块与场景拆分需求。
- 说明触发条件、前置条件、主流程、分支流程、异常流程。
- 涉及数据时补充字段定义与校验规则。

### 4. 定义交互需求

- 将页面状态转换为交互行为描述。
- 说明状态切换、反馈时机、导航行为。
- 覆盖边界交互：重试、权限拒绝、空数据、危险操作确认。

### 5. 定义验收与指标

- 用可测试语句编写验收标准。
- 增加非功能需求（性能、可靠性、可用性、安全基线）。
- 定义成功指标与埋点/观测需求。

### 6. 生成 PRD 输出

- 默认使用 `references/prd-template.md`。
- 需求编号保持稳定（`FR-*`、`NFR-*`、`AC-*`）用于追踪。
- 文末补充未决问题与决策记录。

## 质量标准

- 每条需求保持原子化与可测试。
- 除非有约束，不提前泄露实现细节。
- 文案确保设计、研发、测试可无歧义执行。

## 工作流输出路径

- 提供工作流目录时，输出写入：
- `docs/workflows/<workflow-id>/02-requirements/prd.md`
- `docs/workflows/<workflow-id>/02-requirements/interaction-spec.md`
