# 需求文档生成

将 UI 分析结果整理成可直接指导服务端与客户端开发的详细需求文档。

## 适用场景

- 已经完成页面分析，需要沉淀正式需求文档。
- 需要把功能目标拆成服务端职责和客户端职责。
- 需要统一需求编号、验收标准、接口说明和页面行为说明。

## 工作流

### 1. 统一输入

- 输入优先读取：`01-ui-analysis/pages/*.md` 与 `01-ui-analysis/component-inventory.md`。
- 收集业务目标、角色、约束条件、现有系统限制。
- 将模糊目标改写为可衡量目标，将未知项登记为待确认问题。

### 2. 定义范围与目标

- 写清业务背景、用户问题、价值目标。
- 明确 in-scope 与 out-of-scope。
- 以稳定编号记录功能需求、非功能需求和验收标准。

### 3. 先编写服务端需求

- 定义服务端职责、业务规则、数据模型、接口范围。
- 定义接口输入输出、字段校验、异常返回、幂等与权限要求。
- 标注需要服务端优先完成的能力和依赖关系。

### 4. 再编写客户端需求

- 定义页面功能、交互流程、状态切换、组件行为。
- 对接服务端接口时，明确数据映射、错误反馈、重试与空态策略。
- 约束客户端只承载展示逻辑和交互编排，不承载服务端业务不变式。

### 5. 定义验收与风险

- 用可测试语句编写验收标准。
- 增加性能、可靠性、安全、可用性等非功能要求。
- 记录关键风险、外部依赖、待确认问题与决策建议。

### 6. 输出需求文档

- 使用 `references/prd-template.md` 作为默认模板。
- 服务端需求与客户端需求写在同一套文档中，但分章节表达。
- 若需要补充分离的交互说明，可同时输出交互规格文档。

## 质量标准

- 服务端需求先于客户端需求，顺序不可颠倒。
- 每条需求都应能映射回页面功能点或业务目标。
- 文档内容可被开发阶段直接消费，不需要再次拆解。
- 编号在后续实现和文档中保持稳定。

## 输出路径

- 单独执行时，默认写入：
- `docs/<task-id>/02-requirements/requirements.md`
- `docs/<task-id>/02-requirements/interaction-spec.md`
- 串联执行时，沿用同一 `docs/<task-id>/` 目录。
