# 代码审查与测试审计

在合并或发布前，对代码风险与测试覆盖进行系统审计。

## 工作流

### 1. 确定审查范围

- 明确变更文件、影响业务域、用户可见影响。
- 优先覆盖关键路径：支付、库存、认证、权限、订单生命周期。
- 对上下文缺口先记录假设。

### 2. 执行代码审查

- 识别正确性缺陷、行为回归、边界条件问题。
- 检查是否违反领域边界或架构约束。
- 识别可靠性、安全、性能、可观测性缺口。

### 3. 分级输出问题

- 按严重级别排序：致命、高、中、低。
- 每条问题给出文件/行号证据与触发场景。
- 区分必须修复项与可延期优化项。

### 4. 设计测试策略

- 按业务流与技术层构建风险驱动测试矩阵。
- 明确单元、集成、契约、端到端优先级。
- 增加外部依赖异常下的负向与韧性测试。

### 5. 产出可执行结果

- 使用 `references/review-test-template.md`。
- 输出必须包含：
- 按严重级别排序的问题清单
- 最小修复方案
- 按优先级组织的测试用例
- 未关闭风险说明

## 质量标准

- 以证据驱动结论，避免泛化评论。
- 关注行为和风险，不围绕个人编码偏好。
- 测试计划需可直接分配给研发与测试执行。
