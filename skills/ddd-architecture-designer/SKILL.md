# DDD 架构设计

将需求文档转换为可实施的三端 DDD 架构方案。

## 工作流

### 1. 澄清领域与子域

- 从需求中提取核心域、支撑域与通用能力。
- 建立统一业务词汇表并统一命名。
- 规则不完整处显式标记假设。

### 2. 划分限界上下文

- 按业务一致性与团队所有权划分上下文。
- 定义上下文关系（共享内核、客户-供应商、防腐层、发布语言）。
- 避免仅按技术层做切分。

### 3. 建模领域内部结构

- 定义实体、值对象、聚合、仓储、领域服务、领域事件。
- 明确聚合内部不变式。
- 定义命令/查询职责与事务边界。

### 4. 分配三端职责

- 客户端：展示模型、交互编排、本地缓存策略。
- 服务端：应用服务、领域模型、集成适配、持久化。
- 后台端：运营流程、审计能力、权限治理视图。

### 5. 定义集成契约

- 定义 API 契约、事件契约、版本策略。
- 按用例定义一致性策略（强一致、最终一致、补偿）。
- 定义失败处理与幂等策略。

### 6. 输出架构文档

- 使用 `references/architecture-template.md`。
- 至少包含：上下文地图、模块映射、时序说明、部署要点。
- 与需求编号保持可追踪关系。

## 质量标准

- 每个限界上下文保持语言一致性。
- 模型复杂度与业务复杂度匹配。
- 架构取舍必须显式记录。
